# CONTENT MIGRATION PLAN

Plan for migrating the public content from the old Fresvik Produkt website at
`https://www.fresvik.no/` into the new Next.js App Router + TypeScript +
Tailwind + Sanity project.

The old site is the source for content, links, images, PDFs, and page inventory.
It is not the source for the new visual design. If content or assets are not
confirmed during migration, mark them as `TODO` instead of inventing details.

## 1. Current Project State

### Existing pages and routes

- `/` exists as the current home route.
- `/kontakt` exists and is already wired to Sanity with local fallback content.
- `/studio` exists as the embedded Sanity Studio route.
- `/robots.txt` is generated by `src/app/robots.ts`.
- `/sitemap.xml` is generated by `src/app/sitemap.ts`.
- `/icon.svg` and favicon assets exist.

### Existing components

- `ContactForm` handles the contact form UI on `/kontakt`.
- `SanityStudio` renders the embedded Sanity Studio.
- `StudioShell` wraps the Studio route.

### Existing data and CMS setup

- Local fallback contact data exists in `src/data/contact.ts`.
- Sanity schema for `contactPage` exists in
  `src/sanity/schemaTypes/contactPage.ts`.
- Starter contact data exists in `sanity/seed/contactPage.ndjson`.
- Sanity is configured through:
  - `src/sanity/env.ts`
  - `src/sanity/lib/client.ts`
  - `src/sanity/lib/queries.ts`
  - `src/sanity/lib/contactPage.ts`
  - `sanity.config.ts`
  - `sanity.cli.ts`

### Verification commands

- `npm run lint`
- `npm run build`

### Do not break

- Do not break the existing `/kontakt` route.
- Do not remove the Sanity fallback behavior for contact content.
- Do not commit local secrets or `.env.local`.
- Do not break the Vercel deployment setup.
- Do not break the GitHub SSH workflow.
- Do not replace the new design direction with a one-to-one copy of the old
  site.

## 2. Architecture To Prepare Before Bulk Migration

Before migrating many pages, prepare shared layout and UI foundations so every
new page follows one consistent structure.

### Layout components

- `src/components/layout/Header.tsx`
- `src/components/layout/Footer.tsx`
- `src/components/layout/SiteShell.tsx`

Header and Footer must be shared across all public pages. They should use real
routes only; navigation must not use `href="#"`.

### Base UI components

- `src/components/ui/Container.tsx`
- `src/components/ui/SectionHeader.tsx`
- `src/components/ui/Button.tsx`
- `src/components/ui/Card.tsx`
- `src/components/CTASection.tsx`

### Shared data and config

- `src/data/navigation.ts` for public navigation links.
- `src/config/site.ts` for site name, base URL, contact defaults, and shared SEO
  defaults.

## 6. Assets Migration

### Files to collect

- All public images from the old site.
- Logo files and brand marks.
- Icons that are real content, not purely old design decoration.
- PDF files.
- External document links.
- Alt text or nearby text that can become alt text.
- Social media links.
- SINTEF, Miljøfyrtårn, PDF, and monteringsanvisningar documents.

### Temporary local structure

Use this temporary structure while auditing and rebuilding pages:

- `public/assets/fresvik/images/`
- `public/assets/fresvik/documents/`
- `public/assets/fresvik/logos/`

### Recommended long-term asset strategy

- Temporarily store source assets in `public/assets/fresvik/...` during the
  migration so pages can be built quickly and reviewed.
- Move editor-managed images and documents into Sanity assets after the content
  models are stable.
- Keep only permanent static brand assets in `public`.
- Track missing or unclear assets as `TODO` in the page migration table or in
  the final `MIGRATION_REPORT.md`.

## 8. SEO And Redirects

### SEO requirements

- Add page-level metadata for all migrated pages.
- Add canonical URLs.
- Add OpenGraph title, description, and image where useful.
- Keep `src/app/sitemap.ts` updated as routes are added.
- Keep `src/app/robots.ts` valid for production.
- Add JSON-LD for Organization globally.
- Add JSON-LD for Product on product pages where the content supports it.

### Redirect requirements

- Preserve existing old URLs whenever the new structure uses the same path.
- Add redirects for changed paths, especially:
  - `/produkt/fresvik-panel` to `/produkt/fresvik-pur-panel` if the content
    mapping is confirmed.
- Before launch, crawl the old URL list and verify that no important old link
  returns a dead route on the new site.

## 10. Definition Of Done

The migration is done only when:

- All main pages from the old website exist in the new project or are marked
  with `TODO`.
- Navigation works without `href="#"`.
- Header and Footer are shared.
- `/kontakt` still works.
- The home page is created.
- Product pages are created.
- Service pages are created.
- Documentation and PDFs are migrated or marked `TODO`.
- FAQ is migrated.
- Employees are migrated.
- News/current content is migrated.
- References are migrated.
- `personvernerklering` and `openheitslova` are migrated.
- `sitemap.ts` and `robots.ts` are updated.
- Main pages have metadata.
- Images have alt text.
- `npm run lint` passes.
- `npm run build` passes.
- A final `MIGRATION_REPORT.md` exists with what was migrated, what remains as
  `TODO`, and any content/assets that could not be confirmed.
